# 블로그 글 자동 작성기

## 전체 워크플로우

```
[사진 + 지도링크 + 메모 입력]
  → [지도 링크 분석: 메뉴·위치·영업시간·가격대 추출]
  → [사진 분석: 구도·내용 파악 + 배치 결정]
  → [SEO 최적화 글 작성 (humanizer 내재화)]
  → [사용자 검토/수정 → 완료]
```

글 작성까지만 담당. 네이버 블로그 자동 업로드는 `naver-blog-uploader` 스킬 사용.

---

## 0단계: 기존 작업 확인 (필수)

사진/메모가 들어오면 **바로 저장·작성하기 전에** 먼저 체크:

```bash
ls blog-posts/{장소명}/post_info.md 2>/dev/null
```

`post_info.md`가 존재하면 `status` 필드 확인:

| status | 할 일 |
|--------|-------|
| `images_saved` | 이미지 재저장 없이 → 초안 작성만 |
| `draft_ready` | 초안 그대로 → "이전에 작성된 초안 있어요, 업로드할까요?" 안내 |
| `published` | "이미 발행된 글이에요" 안내 후 종료 |

**이미지 중복 방지**: 저장 전 `post_info.md`의 `source_files` 목록과 비교. 같은 inbound 파일명이면 스킵.

## 1단계: 입력 수집 + 사진 수 확인 (필수)

| 입력 | 필수 | 설명 |
|------|------|------|
| 사진 | ✅ | 1장 이상 |
| 메모/코멘트 | ✅ | 먹은 것, 느낀 점, 특이사항 (자유 형식) |
| 네이버 지도 링크 | 권장 | 영업시간·가격대·메뉴·주차 정보 추출 |

### 🚨 사진 수 확인 프로토콜 (절대 생략 금지)

Telegram은 여러 장 사진을 **배치(batch)로 분할 전송**한다. 사용자가 9장을 보내도 에이전트가 처음 4장만 수신한 상태에서 작업을 시작할 수 있다.

**규칙**:
1. 사진이 첨부된 메시지를 받으면 **즉시 작업 시작 금지**
2. 수신된 사진 수를 세어 사용자에게 먼저 확인 질문:
   > "사진 N장 받았어요. 다 왔나요?"
3. 사용자가 "다 보냈어" / "맞아" / "응" 등으로 확인하면 → **즉시 2~5단계 전체를 one-shot으로 실행** (추가 확인 없이)
4. 수가 다르면: "잠깐 더 기다려볼게요, 아직 M장이 덜 온 것 같아요" → 추가 사진 도착 대기
5. 사용자가 "다 보냈어" / "맞아" 등으로 확인하면 그때 진행

**예외**: 사용자가 메시지에 "사진 N장" 이라고 직접 명시한 경우 → 수신된 수가 N장이면 바로 확인 메시지 없이 진행 가능

> 이 단계를 건너뛰면 일부 사진이 누락된 채 글이 작성된다. 반드시 수 확인 후 진행할 것.

### 🚨 사진 확인 후 행동 원칙 (가장 중요)

사용자가 사진 수를 확인해준 순간 → **글 완성본을 바로 출력**한다.

절대 하면 안 되는 것:
- "이미지 저장 중..." 메시지만 보내고 멈추기
- "지도 분석 중..." 보고만 하고 다음 메시지 기다리기
- 중간 단계 진행상황 보고 후 사용자 응답 대기

올바른 흐름:
```
사용자: "다 보냈어"
에이전트: [이미지 저장 + 지도 fetch + 사진 분석 + 글 작성] → 초안 전체 출력
```
한 번의 응답에 초안 전체가 나와야 한다.

## 2단계: 지도 링크 분석

네이버 지도 링크(`naver.me/...` 또는 `map.naver.com/...`)는 **Puppeteer MCP**로 크롤링한다. `web_fetch`는 JS 렌더링이 안 되므로 사용 금지.

### Puppeteer 크롤링 절차

```
1. mcp__puppeteer__puppeteer_navigate(url, launchOptions: { headless: true }) — 단축 URL 그대로 전달 (리다이렉트 자동 처리, headless 필수)
2. mcp__puppeteer__puppeteer_evaluate — iframe src 추출:
   document.querySelector('iframe[src*="pcmap.place.naver.com"]')?.src
3. mcp__puppeteer__puppeteer_navigate(iframeSrc) — place 상세 페이지로 이동
4. mcp__puppeteer__puppeteer_evaluate — 영업시간 펼치기:
   document.querySelector('[data-v-*]')  // "펼쳐보기" 버튼 클릭
   → 버튼 텍스트가 "펼쳐보기"인 요소 클릭:
     Array.from(document.querySelectorAll('a,button,span')).find(el=>el.innerText.trim()==='펼쳐보기')?.click()
5. mcp__puppeteer__puppeteer_evaluate — 텍스트 추출:
   document.body.innerText
```

아래 정보를 추출해서 본문에 **텍스트로 풀어 설명** (단순 URL 삽입 금지):

| 추출 항목 | 활용 방법 |
|-----------|-----------|
| 장소명 | 제목 키워드 + 본문 자연스럽게 5~8회 |
| 주소 / 지역명 | 제목 앞부분 + 해시태그 |
| 카테고리 | 글 톤 및 해시태그 조정 |
| 메뉴 / 가격대 | 종결부 꿀팁 섹션에 정리 |
| 영업시간 | 종결부 꿀팁 섹션에 정리 (요일별 상세 포함) |
| 주차 정보 | 종결부 꿀팁 섹션에 정리 |

> Puppeteer MCP가 없거나 크롤링 실패 시: 사용자 메모 + 주소 정보로만 작성, 꿀팁 섹션 생략

### Puppeteer MCP 미설치 시 설정 안내

Puppeteer MCP가 없으면 아래 명령으로 설치:

```bash
claude mcp add puppeteer -- npx -y @modelcontextprotocol/server-puppeteer
```

설치 후 Claude Code를 재시작하면 `mcp__puppeteer__*` 도구가 활성화된다.

## 3단계: 사진 분석

`image` 툴로 각 사진 분석 → 배치 결정

**🚨 절대 규칙: 사용자가 첨부한 사진은 단 한 장도 빠짐없이 전부 본문에 [그림N] 마커로 삽입한다.**
- N장 첨부 → 본문에 반드시 [그림1]~[그림N] 전부 등장해야 한다
- 글 완성 후 첨부 사진 수 vs [그림N] 개수를 반드시 대조 검증할 것
- 임의로 제외하거나 "비슷한 구도라 생략" 하는 것 절대 금지
- 나중에 추가 사진이 오면 즉시 해당 위치에 삽입 → 본문을 업데이트해서 다시 보여준다

| 사진 유형 | 배치 위치 |
|-----------|-----------|
| 외관/간판 | 오프닝 직후 |
| 공간/인테리어 | 공간 묘사 단락 직후 |
| 음식 전체 풀샷 | 메뉴 소개 시작 직후 |
| 음식 클로즈업 | 해당 메뉴 묘사 직후 |
| 먹는 중/테이블 현황 | 체험 단락 직후 |

**이미지 묘사 규칙**:
- ❌ "첫 번째 사진은~", "이 사진에서 보시다시피~" 금지
- ✅ "문 열자마자 보이는~", "자리 앉자마자~", "나오자마자~" 처럼 자연스럽게 장면으로 연결

## 4단계: 글 작성

### 4-1. 제목 규칙 (SEO + 클릭률)

**구조**: `[지역명 + 장소유형 + 장소명] | [구체적 메뉴/특징 + 클릭 유발 문구]`

#### 핵심 원칙
1. **지역명은 제목 앞에** — 검색 유입의 핵심. `익선동 카페 카페하이웨스트` > `카페하이웨스트 익선동`
2. **메뉴를 직접 명시** — "케이크 맛집"보단 "레드벨벳 케이크", "스테이크 플래터 26000원"처럼 구체적으로
3. **파이프`|` 뒤에 클릭 유발 문구** — 궁금증, 공감, 숫자, 반전 중 하나 사용
4. **30자 내외 유지** — 모바일 검색 결과에서 잘리지 않도록

#### 클릭 유발 문구 유형 (뒷부분에 활용)
| 유형 | 예시 |
|------|------|
| 웨이팅/화제성 | `주말마다 줄 서는 이유 있는 집`, `30분 웨이팅 각오하고 갈 만한` |
| 구체적 가격/가성비 | `스테이크 플래터 26,000원이면 납득`, `1인 2만원대 코스` |
| 반전/발견 | `쇼핑몰 안에 이런 데가 있었다고`, `골목 끝까지 걸어온 보람` |
| 메뉴 묘사 | `곰 얼굴 케이크랑 테디 스콘 파는`, `마늘 산처럼 쌓여 나오는` |
| 경험 공감 | `익선동 갔다가 여기서 두 시간 보낸 날`, `이 집 때문에 또 잠실 갔다` |

#### 예시 비교
| ❌ 피할 패턴 | ✅ 좋은 패턴 |
|------------|------------|
| `멕시칼리 건대 맛집` | `건대 타코 맛집 멕시칼리 | 30분 웨이팅 각오하고 갈 만한 집` |
| `쌤쌤쌤 롯데월드몰 후기` | `잠실 흑백요리사 셰프 식당 쌤쌤쌤 | 라자냐 먹으러 롯데월드몰 간 날` |
| `종로 닭도리탕` | `종로 노포 맛집 계림 닭도리탕 | 마늘 산처럼 쌓여 나오는 집` |
| `익선동 카페 후기` | `종로 빈티지 카페 카페하이웨스트 | 곰 얼굴 케이크랑 테디 스콘 파는 익선동 숨은 맛집` |
| `르뵈프 롯데월드몰` | `잠실 롯데월드몰 맛집 르뵈프 | 6층 끝까지 걸어온 보람 있는 스테이크 비스트로` |

#### 제목 작성 후 자가검증
- [ ] 지역명이 앞에 있는가?
- [ ] 대표 메뉴 또는 장소 특징이 구체적으로 들어갔는가?
- [ ] 클릭하고 싶게 만드는 문구가 있는가?
- [ ] 30자 내외인가? (파이프 포함 40자 이내)

### 4-2. 본문 구조

```
[도입부]
방문 계기 or 현재 트렌드 언급 → 공감 유도
"요즘 이 동네 핫하다던데", "SNS에서 봤을 때부터 가보고 싶었는데" 류

[그림1] ← 외관 or 첫인상 사진

[중반부 - 공간]
인테리어·분위기 묘사 (감각적, 직접 경험 위주)

[그림2] ← 공간 사진

[중반부 - 음식]
메뉴별 묘사 (단순 나열 X, 먹는 경험 중심)
개인 에피소드 1개 포함

[그림3~N] ← 메뉴 묘사 직후마다

[종결부]
재방문 의사 한 줄 + 꿀팁 요약 박스
```

### 4-3. 종결부 꿀팁 박스

지도 링크 정보를 텍스트로 풀어서 아래 형식으로 정리:

```
📍 [장소명] 방문 전 체크

영업시간 | 월~금 11:00~21:00, 주말 11:00~22:00
웨이팅   | 주말 기준 30분~1시간 예상, 입장 후 실내 대기
주차     | 근처 유료 주차장 이용 (도보 3분)
가격대   | 1인 15,000~25,000원대
```

> 지도 크롤링 실패 시 사용자 메모에서 확인 가능한 정보만 기재, 모르는 항목은 생략

### 4-4. 페르소나
→ `references/persona_20s_female.md` 참고

요약:
- 27세 서울 직장인 여성, 차분하지만 솔직한 말투
- `~해요` / `~구..` / `~잖아요?` 체 혼용
- `..` 줄임표, 괄호 셀프 디스: `(그게 나..)`, `(비밀..)`
- 이모지: 글 전체 1~2개, 문맥에 맞게
  - 허용: ✌️ 🫶 🔥 👀 😅 🤤 😭 🥹 💯 🙏 😮
  - 금지: ✨ ❤️ 🩷
  - 매 글 ✌️ 고정 금지 — 상황에 맞는 걸로

### 4-5. 금지 패턴 (AI 냄새 + SEO 방해)

| 금지 표현/패턴 | 대안 |
|--------------|------|
| `정말 좋았어요`, `너무 맛있어요` | 구체적 감각 묘사로 대체 |
| `강추드려요`, `추천해요` | 재방문 의사로 대체 |
| `겉은 A하고 속은 B한` (대구 병렬) | 자연스럽게 풀어쓰기 |
| `A하지 않고 B한` (부정 연결) | 긍정 표현으로 |
| 쉼표 3개 이상 한 문장 | 문장 분리 |
| `두껍지 않고 얇은` (중복) | `얇고` 하나로 |
| `-들` 복수형 과다 | 삭제하거나 최소화 |
| 번호 매기기 목록 (`1. 2. 3.`) | 산문 형태로 |
| `중요하다`, `핵심`, `효과적` (AI 어휘) | 구어체로 대체 |
| `첫 번째 사진은~`, `이 사진에서~` | 장면 연결 묘사로 |

### 4-6. 좋은 패턴 (사람 냄새 + 체류시간 ↑)

- 짧은 문장 + 긴 문장 리듬 번갈아
- 동사/형용사 직접 사용 (`바삭해서` > `바삭거리는 맛이 있어서`)
- 개인 에피소드 1개 (`(칩 한 접시 거의 다 비운 건 비밀..)`)
- 여운 어미 (`~있구..`, `~됐고요`)
- 단락을 자주 나눠 모바일 가독성 확보
- 키워드(장소명+지역명) 본문에 자연스럽게 5~8회

## 🚨 파이프라인 실행 규칙 (절대 준수)

이미지 저장이 완료되면 **즉시 멈추지 말고 다음 단계로 진행**:

```
이미지 저장 완료
  → 지도 링크 Puppeteer 크롤링 (실패해도 계속)
  → 사진 분석
  → post_info.md 생성 (status: images_saved)
  → 글 작성 (← 여기까지 한 번에 실행, 중간에 사용자 확인 요청 금지)
  → 초안 사용자에게 전달
  → post_info.md status: draft_ready 업데이트
```

**절대 금지**: 이미지 저장 후 "지도 크롤링 중", "사진 분석 중" 등 중간 상태 보고만 하고 멈추는 것. 사용자는 최종 초안을 원한다.

## 5단계: 출력 형식

본문은 복붙 즉시 사용 가능하도록 깨끗하게 출력:

```
제목: [메인키워드] | [감성 문구]

---

[도입부]

[그림1]

[공간 묘사]

[그림2]

[음식 묘사]

[그림3]

...

[클로징 한 줄]

📍 [장소명] 방문 전 체크
영업시간 | ...
웨이팅   | ...
주차     | ...
가격대   | ...

#태그1 #태그2 #태그3 ...
```

초안 출력 후 **반드시** 아래 두 가지 선택지를 제시한다:

```
수정 원하면 말씀해 주세요.
네이버 블로그에 바로 업로드할까요, 아니면 글만 받으실 건가요?
```

사용자 응답에 따라:
- **업로드 원함** ("올려줘", "업로드해줘", "발행해줘" 등) → `naver-blog-uploader` 스킬 사용
- **글만 원함** ("글만", "나중에", "직접 올릴게" 등) → 완료, 추가 액션 없음
- **수정 요청** → 수정 후 다시 위 선택지 제시

⚠️ 사용자의 명시적 확인 없이 업로드 시작 절대 금지

## post_info.md 표준 포맷

이미지 저장 직후, 초안 작성 전에 `post_info.md` 생성. **status 필드 필수**:

```markdown
# {장소명}

## status
images_saved   ← 이미지만 저장됨 (draft 없음)
# draft_ready  ← 초안 작성 완료 (미발행)
# published    ← 발행 완료

## source_files (원본 inbound 경로, 중복 감지용)
- file_91---xxxx.jpg → img1_exterior.jpg
- file_93---xxxx.jpg → img2_interior.jpg

## 입력 메모
...

## 지도 링크
...

## 사진 (총 N장)
- img1_xxx.jpg → 설명
...

## 작성된 글
(draft_ready 이후 여기에 저장)

## 발행 정보
- logNo: (미발행)
- URL: (미발행)
```

**status 업데이트 시점**:
- 이미지 저장 완료 → `images_saved`
- 초안 사용자에게 보여줌 → `draft_ready`
- 발행 완료 → `published` + logNo/URL 기록
